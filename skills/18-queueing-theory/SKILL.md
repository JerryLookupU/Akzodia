# 18 Queueing Theory

## Book-Derived Essence

- Core framework: Arrival process, service process, servers, utili

zation, waiting time, queue length, Little’s Law, and capacity tradeoffs.
- Deep idea: Queues diagnose delay as a structural consequence of variability and utilization, not merely worker slowness.
- Discovery method: Measure arrivals, service times, concurrency, utilization, blocking, priority, retries, and abandonment; look for high utilization, burstiness, and feedback retries.
- Boundary: Do not apply steady-state formulas blindly to adversarial, nonstationary, or strongly coupled workloads.
- Source capsule: `references/source-notes.md#BDE-core-framework`

## When To Use

Use this skill when orchestration performance depends on how work waits for scarce service capacity.

Strong triggers:
- Jobs, prompts, tool calls, reviews, test runs, or subagent tasks arrive over time and may wait in a queue.
- The user asks how many workers, tool slots, browser sessions, API permits, or reviewer lanes are needed.
- Backlog, latency, timeouts, retry storms, rate-limit errors, or SLA misses are symptoms.
- The orchestrator must choose queue disciplines: FIFO, priority, aging, separate queues, admission control, batching, or preemption.
- A design needs explicit utilization, throughput, wait-time, queue-length, and idle/busy-time metrics.
- The system looks "average-capacity sufficient" but still suffers long waits during bursts or near-full utilization.

Prefer scheduling theory when every job is already known and the main issue is ordering. Use queueing theory when jobs arrive dynamically and capacity must absorb uncertainty.

## Activation And Execution Gate

Activate only when dynamic arrivals, waiting, backlog, service capacity, utilization, retries, rate limits, or queue disciplines drive orchestrator performance. ## Workflow

1. Define the queueing boundary.
   - Name the service system: agent pool, tool adapter, test runner, review queue, API gateway, retrieval service, or retry lane.
   - Specify what counts as an arrival, when service starts, when service completes, and whether failures re-enter the queue.
   - Identify servers: workers, tool slots, API permits, browser instances, database connections, human reviewers, or budget-gated execution lanes.
   - Record queue capacity, queue discipline, priority classes, timeout/abandonment rules, cancellation rules, and retry policy.

2. Measure arrivals and service.
   - Estimate arrival rate `lambda` per task class over the same time window used for service measurements.
   - Estimate service rate `mu` per server from completed work, or use service-time percentiles when variance is high.
   - Count effective parallel servers `c`, not nominal workers. Exclude workers blocked by locks, quotas, approvals, shared state, or serial bottlenecks.
   - Separate exogenous arrivals from retries. Retries are load generated by the system and can turn a small incident into congestion.
   - If measurements are weak, use ranges and scenarios instead of a single confident number.

3. Choose the model fidelity.
   - Start with Little's Law for steady flow sanity checks: average work in system equals throughput times average time in system.
   - Use utilization `rho = lambda / (c * mu)` as a first congestion signal for one task class served by homogeneous workers.
   - Use simple `M/M/c`-style approximations only when Poisson-like arrivals, exponential-like service, independent jobs, and steady-state assumptions are acceptable.
   - Use simulation or scenario testing when arrivals are bursty, service times are heavy-tailed, priorities interact, retries feed back, or the workflow is nonstationary.
   - If exact probability distributions are unavailable, still use means, variances, and percentiles to compare policies.

4. Find the operating region.
   - Compute current and forecast utilization by task class and shared bottleneck.
   - Treat high utilization as risky even when average throughput is acceptable; wait time rises sharply as utilization approaches full capacity.
   - Check burst cases: deployment spikes, batch imports, incident retries, dependency outages, nightly jobs, and user-facing peak windows.
   - Identify the binding bottleneck: worker count, API quota, human approval, test environment, lock contention, memory, cost budget, or serialized state update.
   - Decide target headroom explicitly, such as keeping normal utilization below a chosen threshold and reserving capacity for urgent work.

5. Convert analysis into orchestration policy.
   - Size workers or permits per queue based on target utilization, wait-time target, and worst credible burst.
   - Split queues when urgent work, cheap work, long-running work, or failure recovery should not wait behind ordinary jobs.
   - Add backpressure: admission limits, rate shaping, queue caps, degradation modes, batching, or "try later" responses.
   - Control retries with caps, jitter, cooldown, circuit breakers, and retry budgets so failed service does not multiply arrivals.
   - Use priority with fairness safeguards: aging, reserved slots, starvation checks, or bounded preemption.
   - Prefer adding capacity only after verifying the bottleneck is really service capacity rather than lock contention, dependency latency, or bad retry policy.

6. Instrument and replan.
   - Monitor arrival rate, service time, queue length, waiting time, sojourn time, utilization, idle fraction, timeout rate, retry rate, and abandonment/cancellation.
   - Track metrics per task class and per bottleneck; aggregate averages can hide priority inversion and starvation.
   - Set triggers for policy changes: queue length above threshold, wait-time percentile breach, utilization above target, retry-rate spike, quota exhaustion, or capacity loss.
   - Report uncertainty and assumptions with the recommendation, especially stationarity, independence, service-time distribution, and retry feedback.

## Output Format / Deliverables

Return a queueing analysis contract:
- Boundary: arrival event, service start/end, servers, effective capacity, queue discipline, capacity limits, timeouts, abandonment, retries, and task classes.
- Measurements: `lambda`, `mu`, `c`, utilization, service-time percentiles, retry load, burst scenarios, and confidence/assumptions.
- Model choice: Little's Law sanity check, utilization estimate, M/M/c-style approximation, simulation/scenario plan, or measurement-first plan.
- Policy: worker/permit sizing, queue split, priority/fairness rules, backpressure, retry budgets, batching, admission control, and degradation modes.
- Monitoring: queue length, wait/sojourn time, utilization, idle fraction, retries, timeouts, abandonment, SLA percentiles, and replan triggers.

## Failure / Recovery / Idempotency

If telemetry is missing, begin with instrumentation and bounded assumptions rather than a precise capacity claim. If formulas do not fit bursty, heavy-tailed, priority, or retry-feedback behavior, switch to simulation or scenario analysis.

For repeated reviews, preserve stable queue names, task-class labels, measurement windows, and policy thresholds so trends are comparable. Recompute recommendations from the latest measurements and report deltas from the prior policy.

When the queue destabilizes, recover by throttling arrivals, pausing or capping retries, isolating priority work, shedding optional work, and checking the true bottleneck before adding workers.

## Hard Rules

- Do not use nominal concurrency when effective service capacity is lower due to locks, quotas, approvals, or shared bottlenecks.
- Do not report exact worker counts without assumptions, target utilization or wait time, and replan triggers.
- Do not ignore retry traffic when estimating arrival rate.
- Do not apply steady-state formulas to bursts, incidents, ramps, or rapidly changing capacity without caveats.
- Do not add capacity before checking whether the bottleneck is service capacity, lock contention, dependency latency, or retry policy.

## Failure Modes

- Treating queueing output as exact truth. Queueing models simplify reality and often provide approximate indicators, not guarantees.
- Using steady-state formulas during startup bursts, incidents, traffic ramps, scheduled batch waves, or rapidly changing capacity.
- Assuming more workers always fixes latency. A shared lock, API quota, database bottleneck, or human approval gate may be the real server.
- Ignoring retry load. Retries can raise the effective arrival rate after service quality drops.
- Optimizing average wait while missing tail latency, priority starvation, or urgent-job SLA failure.
- Running every worker near full utilization. A system with no slack becomes fragile and can build long queues after small disturbances.
- Mixing unrelated task classes in one queue without checking whether long jobs block short or urgent jobs.
- Using nominal concurrency instead of effective service capacity after rate limits, locks, warmup, context switching, and cleanup are included.
- Reporting worker counts without the monitoring and backpressure rules needed to keep the queue stable.

## Boundaries

Queueing theory helps estimate throughput, waiting, queue length, utilization, busy/idle time, and capacity headroom. It does not decide business value, task semantics, safety policy, or which work should be allowed.

Closed-form results often depend on simplified assumptions such as steady state, independent arrivals, and convenient interarrival or service-time distributions. When those assumptions are weak, use queueing concepts for structure, then validate with measurement, simulation, load tests, and conservative headroom.

This skill complements Operations Research and Scheduling Theory. Use Operations Research for broader constrained allocation and objective tradeoffs; use Scheduling Theory for ordering a known set of jobs; use Queueing Theory for dynamic arrivals, waiting, backlog, capacity sizing, and congestion control.
