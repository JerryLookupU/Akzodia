# Candidate Principles

## P1: Successful Standards Usually Clarify Proven Interfaces

Accepted. The standards essay distinguishes successful clarifications from failed proactive scratch designs.

## P2: Proactive Official Standards Can Accelerate Simpler Rivals

Accepted. PL/I, Algol 68, Ada, OS/2, OSI, and the Linux API prediction all support the pattern.

## P3: Primary Architecture Must Precede Secondary Facilities

Accepted. Memo 125 makes this the central diagnosis of COLBY.

## P4: Slogans Must Be Converted Into Precise Principles

Accepted. "Display oriented" and "transaction oriented" are used as examples of dangerous slogans that point to secondary features instead of primary requirements.

## P5: Optimize Only After A Working Baseline Exists

Accepted. Memo 125 states "Make it run before you make it faster" in the context of system ops and microcode.

## P6: A Migration Path Is A Product Requirement, Not A Cleanup Task

Accepted. Memo 125 explicitly says smooth migration must have top priority and proposes System/375/interim products.

## P7: Expert Users Need Performance Feedback

Accepted. One-level store discussion argues that complex system programmers need feedback about what hardware is doing to large workloads.

## P8: Late System Integration Is A Symptom Of Non-Evolutionary Design

Accepted. Memo 125 warns that interactions increase faster than line counts and that traumatic integration should not occur.

## P9: Small Expert Architecture Groups Beat Large Premature Architecture Committees

Accepted with boundary. Memo 125 cites Fred Brooks and recommends a small group for primary architecture. This is not a universal staffing law; it applies when primary architecture is still undefined.

## P10: Security Must Not Prevent Necessary Whole-System Understanding

Accepted. Memo 125 says designers and implementers of secondary components need access to primary architecture documentation.
